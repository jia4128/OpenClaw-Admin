## P1 - 短期实施需求 (1 周内完成)

### 需求 1: 会话持久化到数据库

#### 1.1 需求背景
当前系统的会话信息存储在内存中，服务重启后会导致所有用户会话丢失，影响用户体验和系统可用性。需要将会话持久化到数据库，实现会话的持久存储和恢复。

#### 1.2 功能需求

**FR-1.2.1 会话存储**
- 系统应将用户会话信息存储到数据库中
- 支持会话的创建、读取、更新、删除操作
- 会话数据应包括：用户 ID、会话令牌、创建时间、过期时间、最后活跃时间等

**FR-1.2.2 会话恢复**
- 服务重启后能从数据库恢复有效会话
- 自动过滤已过期会话
- 支持会话状态的实时同步

**FR-1.2.3 会话管理**
- 提供会话查询接口（按用户、按时间范围）
- 支持主动注销会话
- 支持批量清理过期会话

#### 1.3 非功能需求

**NFR-1.3.1 性能**
- 会话读写响应时间 < 50ms
- 支持并发会话数 ≥ 10,000
- 数据库查询优化，建立合适的索引

**NFR-1.3.2 可靠性**
- 会话数据持久化成功率 ≥ 99.9%
- 数据库故障时应有降级方案（如内存缓存）

**NFR-1.3.3 安全性**
- 会话令牌加密存储
- 防止会话固定攻击
- 敏感信息脱敏处理

#### 1.4 验收标准

| 编号 | 验收标准 | 验证方法 |
|------|----------|----------|
| AC-1.4.1 | 用户登录后，会话信息成功写入数据库 | 查询数据库验证会话记录存在 |
| AC-1.4.2 | 服务重启后，已登录用户无需重新登录 | 重启服务后验证会话有效性 |
| AC-1.4.3 | 过期会话自动失效，无法访问受保护资源 | 等待会话过期后尝试访问 |
| AC-1.4.4 | 主动注销后，会话从数据库删除且无法使用 | 调用注销接口后验证会话无效 |
| AC-1.4.5 | 会话读写性能满足要求（<50ms） | 性能测试工具压测 |
| AC-1.4.6 | 数据库故障时系统能降级运行 | 模拟数据库故障验证 |

#### 1.5 技术要点
- 数据库选型：MySQL/PostgreSQL（根据现有架构）
- 会话存储表结构设计
- Redis 缓存层（可选，提升性能）
- 会话序列化/反序列化机制

### 需求 2: 日志脱敏机制

#### 2.1 需求背景
系统日志中可能包含敏感信息（如用户密码、手机号、身份证、银行卡号等），存在信息泄露风险。需要建立日志脱敏机制，确保敏感信息在日志中不被明文记录。

#### 2.2 功能需求

**FR-2.2.1 敏感数据识别**
- 自动识别常见敏感数据类型：
  - 手机号
  - 身份证号
  - 银行卡号
  - 邮箱地址
  - 密码
  - 身份证号
  - 地址信息
- 支持自定义敏感字段配置

**FR-2.2.2 脱敏规则**
- 提供多种脱敏策略：
  - 部分掩码（如：138****1234）
  - 完全替换（如：[已脱敏]）
  - 哈希处理
- 支持按日志级别配置脱敏策略
- 支持不同环境（开发/测试/生产）不同脱敏强度

**FR-2.2.3 日志拦截**
- 在日志输出前自动拦截并脱敏
- 支持 JSON 格式日志的字段级脱敏
- 支持异常堆栈信息中的敏感数据脱敏

#### 2.3 非功能需求

**NFR-2.3.1 性能**
- 脱敏处理对日志性能影响 < 5%
- 支持高并发日志写入

**NFR-2.3.2 可配置性**
- 脱敏规则可通过配置文件动态调整
- 支持热更新脱敏策略

**NFR-2.3.3 兼容性**
- 兼容现有日志框架（如 Winston、Log4j 等）
- 不影响日志的检索和分析功能

#### 2.4 验收标准

| 编号 | 验收标准 | 验证方法 |
|------|----------|----------|
| AC-2.4.1 | 手机号在日志中显示为部分掩码（如 138****1234） | 查看日志文件验证 |
| AC-2.4.2 | 身份证号在日志中完全脱敏 | 查看日志文件验证 |
| AC-2.4.3 | 密码在日志中不出现明文 | 尝试登录并查看日志 |
| AC-2.4.4 | 银行卡号在日志中脱敏 | 查看包含卡号的日志 |
| AC-2.4.5 | 脱敏配置可动态调整 | 修改配置后验证效果 |
| AC-2.4.6 | 脱敏处理不影响日志性能 | 性能对比测试 |
| AC-2.4.7 | 异常堆栈中的敏感信息被脱敏 | 触发异常查看日志 |

#### 2.5 技术要点
- 正则表达式匹配敏感数据
- 日志拦截器/过滤器实现
- 脱敏策略配置中心
- 环境差异化配置

### 需求 3: 环境变量密码哈希

#### 3.1 需求背景
当前系统中部分敏感配置（如数据库密码、API 密钥）以明文形式存储在环境变量或配置文件中，存在安全风险。需要对密码类配置进行哈希处理或加密存储。

#### 3.2 功能需求

**FR-3.2.1 密码检测**
- 自动检测配置中的密码类字段
- 识别常见的密码变量名（如 PASSWORD、SECRET、KEY 等）
- 扫描环境变量和配置文件

**FR-3.2.2 哈希/加密处理**
- 支持密码的哈希存储（如 bcrypt、argon2）
- 支持对称加密存储（如 AES）
- 提供密码加密工具命令行

**FR-3.2.3 运行时解密**
- 应用运行时自动解密密码
- 解密过程在内存中进行，不写入日志
- 支持密钥管理（如使用 KMS 或环境变量存储密钥）

**FR-3.2.4 配置管理**
- 提供加密配置的管理界面或接口
- 支持密码轮换（定期更换）
- 记录密码变更审计日志

#### 3.3 非功能需求

**NFR-3.3.1 安全性**
- 哈希算法强度满足安全要求（bcrypt cost ≥ 10）
- 加密密钥安全存储
- 防止密钥泄露

**NFR-3.3.2 可用性**
- 密码解密对应用透明
- 支持故障恢复机制

**NFR-3.3.3 兼容性**
- 兼容现有配置加载机制
- 支持向后兼容（旧明文配置可迁移）

#### 3.4 验收标准

| 编号 | 验收标准 | 验证方法 |
|------|----------|----------|
| AC-3.4.1 | 环境变量中的密码以哈希/加密形式存储 | 检查环境变量值 |
| AC-3.4.2 | 应用启动时能正确解密并使用密码 | 启动应用验证连接 |
| AC-3.4.3 | 日志中不出现明文密码 | 查看启动日志 |
| AC-3.4.4 | 提供密码加密工具 | 运行加密工具验证 |
| AC-3.4.5 | 支持密码轮换操作 | 执行密码更换流程 |
| AC-3.4.6 | 密码变更有审计日志 | 查看审计日志 |
| AC-3.4.7 | 密钥安全存储，不硬编码在代码中 | 代码审查 |

#### 3.5 技术要点
- bcrypt/argon2 哈希算法
- AES-256 加密
- 密钥管理系统
- 配置加密工具开发

## P2 - 中期优化需求 (1 个月内完成)

### 需求 4: 双因素认证 (2FA)

#### 4.1 需求背景
仅依靠用户名密码的认证方式存在被暴力破解、钓鱼攻击等风险。需要引入双因素认证，提升账户安全性。

#### 4.2 功能需求

**FR-4.2.1 认证方式**
- 支持 TOTP（基于时间的一次性密码，如 Google Authenticator）
- 支持短信验证码
- 支持邮箱验证码
- 支持硬件密钥（如 YubiKey，可选）

**FR-4.2.2 用户启用**
- 用户可在账户设置中启用/禁用 2FA
- 提供 QR 码扫描配置（TOTP）
- 提供备用验证码（recovery codes）
- 引导用户完成配置流程

**FR-4.2.3 认证流程**
- 登录时先验证用户名密码
- 再验证第二因素（验证码）
- 支持"信任此设备"选项（可选）
- 支持备用验证码登录

**FR-4.2.4 管理功能**
- 管理员可强制要求某些角色启用 2FA
- 查看用户 2FA 启用状态
- 协助用户重置 2FA（需审批流程）

#### 4.3 非功能需求

**NFR-4.3.1 用户体验**
- 2FA 验证流程简洁，不超过 3 步
- 提供清晰的配置指引
- 备用方案易于获取

**NFR-4.3.2 安全性**
- TOTP 时间窗口合理（±1 分钟）
- 验证码尝试次数限制
- 防重放攻击

**NFR-4.3.3 可用性**
- 支持多种 2FA 方式，避免单点故障
- 备用验证码机制完善

#### 4.4 验收标准

| 编号 | 验收标准 | 验证方法 |
|------|----------|----------|
| AC-4.4.1 | 用户可启用 TOTP 双因素认证 | 用户配置流程测试 |
| AC-4.4.2 | 登录时需输入第二因素验证码 | 登录流程测试 |
| AC-4.4.3 | 验证码错误时给出适当提示（不泄露信息） | 错误场景测试 |
| AC-4.4.4 | 提供备用验证码，可在失去 2FA 设备时使用 | 备用码验证测试 |
| AC-4.4.5 | 支持短信/邮箱验证码方式 | 多种 2FA 方式测试 |
| AC-4.4.6 | 管理员可强制要求 2FA | 管理员配置测试 |
| AC-4.4.7 | 验证码尝试次数有限制，防止暴力破解 | 安全测试 |
| AC-4.4.8 | "信任设备"功能正常工作 | 功能测试 |

#### 4.5 技术要点
- OTP 库集成（如 speakeasy、otplib）
- QR 码生成
- 短信/邮件服务集成
- 备用验证码生成与管理

### 需求 5: WAF 规则实施

#### 5.1 需求背景
系统需要防护常见的 Web 攻击（如 SQL 注入、XSS、命令注入等），需要在应用层部署 Web 应用防火墙（WAF）规则。

#### 5.2 功能需求

**FR-5.2.1 攻击防护**
- SQL 注入防护
- XSS（跨站脚本）防护
- CSRF（跨站请求伪造）防护
- 命令注入防护
- 路径遍历防护
- 文件包含防护
- SSRF（服务端请求伪造）防护

**FR-5.2.2 规则管理**
- 提供规则配置界面或配置文件
- 支持自定义规则
- 支持规则启用/禁用
- 支持规则优先级设置

**FR-5.2.3 威胁检测**
- 检测并记录可疑请求
- 识别攻击模式
- 自动封禁高频攻击 IP
- 提供威胁情报集成（可选）

**FR-5.2.4 日志与告警**
- 记录所有被拦截的请求
- 提供攻击统计报表
- 支持实时告警（如邮件、 webhook）

#### 5.3 非功能需求

**NFR-5.3.1 性能**
- WAF 规则匹配对请求延迟影响 < 10ms
- 支持高并发请求处理

**NFR-5.3.2 准确性**
- 误报率 < 0.1%
- 漏报率 < 0.01%

**NFR-5.3.3 可维护性**
- 规则易于更新和维护
- 提供规则测试工具

#### 5.4 验收标准

| 编号 | 验收标准 | 验证方法 |
|------|----------|----------|
| AC-5.4.1 | SQL 注入攻击被拦截并记录 | 渗透测试 |
| AC-5.4.2 | XSS 攻击被拦截并记录 | 渗透测试 |
| AC-5.4.3 | 命令注入攻击被拦截 | 渗透测试 |
| AC-5.4.4 | 路径遍历攻击被拦截 | 渗透测试 |
| AC-5.4.5 | 高频攻击 IP 被自动封禁 | 模拟攻击测试 |
| AC-5.4.6 | 被拦截请求有详细日志记录 | 查看日志 |
| AC-5.4.7 | 提供攻击统计报表 | 查看报表 |
| AC-5.4.8 | 自定义规则生效 | 配置自定义规则测试 |
| AC-5.4.9 | WAF 对正常请求性能影响 < 10ms | 性能测试 |
| AC-5.4.10 | 误报率满足要求 | 压力测试统计 |

#### 5.5 技术要点
- WAF 引擎选型（如 Express-WAF、Node-WAF 或自研）
- 规则引擎设计
- 正则表达式优化
- IP 封禁机制
- 威胁情报集成

### 需求 6: CI/CD安全扫描集成

#### 6.1 需求背景
在软件开发生命周期中引入安全扫描，实现"安全左移"，在代码提交、构建、部署等阶段自动进行安全检测，及时发现和修复安全问题。

#### 6.2 功能需求

**FR-6.2.1 代码安全扫描**
- 静态应用安全测试（SAST）
- 检测常见安全漏洞（OWASP Top 10）
- 检测硬编码密钥、密码
- 检测不安全的使用方式

**FR-6.2.2 依赖安全扫描**
- 软件成分分析（SCA）
- 检测第三方依赖的已知漏洞
- 检测许可证风险
- 提供依赖更新建议

**FR-6.2.3 容器安全扫描**
- 容器镜像漏洞扫描
- 检测不安全配置
- 检测敏感信息泄露

**FR-6.2.4 CI/CD集成**
- 集成到现有 CI/CD 流程（如 GitHub Actions、GitLab CI）
- 扫描结果自动反馈给开发者
- 支持质量门禁（安全不达标阻断发布）
- 生成安全报告

**FR-6.2.5 漏洞管理**
- 漏洞跟踪和修复建议
- 漏洞优先级排序
- 修复进度统计

#### 6.3 非功能需求

**NFR-6.3.1 扫描速度**
- 代码扫描时间 < 5 分钟（中等规模项目）
- 依赖扫描时间 < 2 分钟
- 容器扫描时间 < 3 分钟

**NFR-6.3.2 准确性**
- 漏洞检测准确率 > 95%
- 误报率 < 5%

**NFR-6.3.3 易用性**
- 扫描配置简单
- 报告清晰易懂
- 修复建议具体可操作

#### 6.4 验收标准

| 编号 | 验收标准 | 验证方法 |
|------|----------|----------|
| AC-6.4.1 | 代码提交时自动触发 SAST 扫描 | 提交代码验证 |
| AC-6.4.2 | 扫描发现安全漏洞并报告 | 引入漏洞测试 |
| AC-6.4.3 | 依赖扫描能识别已知漏洞 | 引入有漏洞的依赖测试 |
| AC-6.4.4 | 容器镜像扫描正常工作 | 构建镜像后扫描 |
| AC-6.4.5 | 安全不达标时阻断发布 | 模拟安全失败场景 |
| AC-6.4.6 | 扫描报告包含修复建议 | 查看报告 |
| AC-6.4.7 | 扫描结果集成到代码平台（如 PR 评论） | 查看 PR 评论 |
| AC-6.4.8 | 扫描时间满足要求 | 计时测试 |
| AC-6.4.9 | 误报率在可接受范围 | 统计误报数量 |
| AC-6.4.10 | 支持自定义安全规则 | 配置自定义规则测试 |

#### 6.5 技术要点
- SAST 工具选型（如 SonarQube、Semgrep、CodeQL）
- SCA 工具（如 Dependabot、Snyk、OWASP Dependency-Check）
- 容器扫描工具（如 Trivy、Clair）
- CI/CD 流水线配置
- 质量门禁策略
